고민의 흔적

기술 의사결정 6장면

기술 나열보다 트레이드오프가 있는 결정이 실력을 보여준다고 생각합니다. 전체 27건은 저장소의 결정로그에 있습니다.

결정 ①

알림은 FCM이 아니라 로컬로

루틴 시간은 기기가 이미 아는 정보인데 서버가 푸시할 이유가 없었습니다. Firebase 프로젝트·토큰 관리·스케줄러를 전부 제거했습니다.

오프라인 동작 + 유지보수 표면적 최소화. 서버 푸시는 리인게이지먼트 용도로만 로드맵에 남김.

결정 ②

N+1은 리뷰가 아니라 구조로 막는다

전 연관관계 LAZY + batch_fetch_size 전역 설정, 조회 화면은 DTO/id 프로젝션. "오늘의 루틴+완료 여부"는 쿼리 2번, 캘린더는 group by 1번.

소유권 검증도 쿼리 조건으로 처리해 지연 로딩 프록시를 건드리지 않음. OSIV off로 피크 트래픽 대비.

결정 ③

AI 기능을 LLM 없이 먼저 출시 가능하게

생성기를 인터페이스로 추상화하고 v1은 룰 엔진으로 전 테마(여행·운동·식단·독서·공부·세안) 구현 — 원가 0원. 세안은 성분 궁합 규칙(레티놀·비타민C·AHA 요일 분리)까지.

LLM 키 확보 전에도 기능이 완결. LLM은 같은 인터페이스의 구현체로 추가 — 호출부 무수정.

결정 ④

실패 과금은 현금이 아니라 재화로

집중 모드 포기, AI 플랜 사용은 앱 내 재화(조개)로만 결제. 유사투자자문업 규제로 첫 아이디어(유료 주식 추천)를 기각한 경험이 원칙이 됐습니다.

사용자가 약해진 순간의 현금 과금은 브랜드 리스크. Duolingo·Forest의 검증된 커밋먼트 디바이스 패턴.

결정 ⑤

도장 쾅의 손맛은 3중으로

Lottie 애니메이션(흰 배경 PNG를 픽셀 단위 투명 처리 스크립트로 구제) + 햅틱 + 코드로 직접 합성한 효과음(95Hz 감쇠 사인파, 19KB, 라이선스 청정).

에셋 렌더 지점을 컴포넌트 1곳으로 단일화 — 이모지→일러스트→Lottie 교체가 파일 1개 수정.

결정 ⑥

브라우저 워크스루로 CORS 결함 발견

출시 전 점검에서 실제 클릭 검증을 돌리자 웹 빌드의 모든 API가 CORS로 차단되고 있음을 발견 — API 도구 테스트만으로는 영원히 안 보였을 결함.

"코드 리뷰 통과 ≠ 사용자 화면에서 동작"을 체득. 이후 기능마다 실구동 검증을 기본 절차로.
