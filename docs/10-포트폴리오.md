# 10. 포트폴리오 — 기술 의사결정과 문제 해결 기록

> **작성 규칙**: 결정로그(03)는 "무엇을 정했나"의 사실 기록, 이 문서는 "왜, 어떻게, 그래서"의 스토리 기록.
> 각 항목은 **상황 → 고민(선택지) → 결정 → 근거 → 결과/배움** 구조로 쓴다. 이 구조 그대로 면접 답변과 자소서 문단이 된다.
> 마일스톤마다 갱신하고, 수치(쿼리 수, 테스트 수, 시간)를 남길 수 있으면 반드시 남긴다.

## 프로젝트 개요 (자소서 첫 문단용)

- **Sealo (씰로)**: 물범이 도장을 찍어주는 루틴 관리 앱. Seal(물범)=Seal(도장) 언어유희 기반 글로벌 지향 컨셉
- 기간: 2026.07 ~ (진행 중) / 1인 풀스택
- 스택: Spring Boot 4.1 (Java 21, JPA, PostgreSQL) + React Native (Expo, TypeScript, TanStack Query)
- 저장소: github.com/EunhoRoh/sealo

## A. 기획·아키텍처 의사결정

### A-1. 유료 주식 추천 앱 아이디어를 규제 리스크로 기각
- **상황**: 첫 후보는 "1,000원에 AI 추천 종목 3개" 주식 앱
- **고민**: 수익 모델은 명확하지만, 대가를 받는 종목 추천은 자본시장법상 유사투자자문업 신고 대상
- **결정**: 아이디어 기각, 루틴 앱으로 전환
- **배움**: 기술 검토 전에 법적·규제 검토가 먼저다. "만들 수 있는가"보다 "출시할 수 있는가"

### A-2. 푸시 알림을 FCM이 아닌 로컬 알림으로
- **상황**: 루틴 시간 알림 기능. 처음 계획은 FCM 서버 푸시
- **고민**: FCM은 Firebase 프로젝트, 디바이스 토큰 관리, 서버 스케줄러가 전부 필요. 그런데 루틴 시간은 기기가 이미 알고 있는 정보
- **결정**: expo-notifications 로컬 주간 알림. FCM은 리인게이지먼트(서버가 판단하는 알림)용으로만 v2에 도입
- **결과**: 서버 인프라 3종 제거, 오프라인 동작 확보. 알림 동기화는 "전체 취소 후 재등록" 멱등 전략으로 diff 관리 로직 자체를 없앰
- **배움**: 기능 요구("알림")와 기술 수단("푸시")을 분리해서 생각하면 아키텍처가 단순해진다

### A-3. 데이터 기반 수익화 설계 (리서치 → MVP 역산)
- **상황**: 광고/구독/인앱결제 중 무엇을 언제 도입할지
- **근거 수집**: RevenueCat(11.5만 앱)·Adapty 리포트 교차 검증 — Health&Fitness는 연간 구독이 매출 57~68%, 주간 플랜은 30일 내 65% 이탈, 하이브리드 수익화는 전체 앱의 ~10%
- **결정**: MVP 무과금(단, 재화 경제는 MVP부터 탑재) → 보상형 광고 → 연간 중심 구독. 핵심 루프에는 영원히 과금 금지 (Duolingo·Finch 공식)
- **배움**: 수익화는 나중에 "붙이는" 게 아니라 경제 구조를 처음부터 설계해야 한다

## B. 성능 설계 (백엔드 면접 단골)

### B-1. N+1을 코드 리뷰가 아니라 구조로 방지
- 모든 연관관계 LAZY + `default_batch_fetch_size: 100` 전역 설정
- 조회 화면은 DTO/id 프로젝션: "오늘의 루틴+완료 여부"는 **쿼리 2번**(루틴 목록 + 도장 id Set), 캘린더는 group by **1번**
- 소유권 검증을 `findByIdAndMemberId` 쿼리 조건으로 — 지연 로딩 프록시를 건드리지 않음
- `open-in-view: false` — 커넥션을 응답 직렬화까지 잡고 있으면 트래픽 피크에 풀 고갈

### B-2. 착용 아이템 전환을 벌크 UPDATE 한 방으로
- **상황**: 모자 착용 시 같은 카테고리 기존 착용 해제 — 조회 후 루프 저장이면 보유 아이템 수만큼 UPDATE
- **결정**: JPQL 벌크 UPDATE 1회 + 대상만 dirty checking

## C. 트러블슈팅 기록

### C-1. Spring Boot 4 마이그레이션 이슈 (2026-07-07)
- **증상**: 테스트 컴파일 실패 — `AutoConfigureMockMvc`, `ObjectMapper` 심볼 없음
- **원인 추적**: Gradle 캐시의 jar를 직접 열어 확인 → Boot 4에서 MockMvc 자동설정이 `org.springframework.boot.webmvc.test.autoconfigure`로 모듈 분리, Jackson이 2.x(`com.fasterxml`)에서 3.x(`tools.jackson`)로 교체됨
- **배움**: 메이저 버전 프레임워크는 블로그가 아닌 실제 클래스패스로 확인하는 게 가장 빠르다

### C-2. JPQL 벌크 UPDATE와 dirty checking의 함정
- **증상(코드 리뷰에서 사전 발견)**: `@Modifying(clearAutomatically=true)` 뒤에 기조회 엔티티의 상태 변경 → 영속성 컨텍스트가 비워져 detached 상태라 변경이 저장 안 됨
- **해결**: clear 없이 벌크 후 대상 엔티티 dirty checking으로 최종 상태 확정, 주석으로 이유 명문화

### C-3. Gradle 툴체인으로 로컬 JDK 불일치 해결
- **상황**: 로컬 JDK 17, 프로젝트는 Java 21
- **해결**: foojay-resolver 플러그인으로 빌드 시 JDK 21 자동 프로비저닝 — 팀원(미래의 나)의 로컬 환경과 무관하게 재현 가능한 빌드

### C-4. 회사/개인 GitHub 계정 충돌
- **증상**: push 시 "Repository not found" — 전역 git 자격증명이 회사 계정
- **해결**: 저장소 로컬로 user.name/email(개인 noreply) + credential.helper 체인 초기화 후 `gh auth git-credential`만 사용

## D. 유지보수 설계

- **에셋 교체 지점 단일화**: 캐릭터/도장 렌더는 컴포넌트 1개(seal-character.tsx, stamp-splash.tsx)에서만. 서버는 `acc_straw_hat` 논리 키만 저장 → 이모지→일러스트→Lottie 교체가 서버 무관, 파일 1개 수정
- **디자인 토큰**(sealo-theme.ts): 색/간격/테두리 하드코딩 금지
- **정책 값 상수화**: 도장 보상(SHELLS_PER_STAMP) 등 밸런스 값은 한 곳에
- **결정로그 운영**: 모든 결정을 번복 이력까지 문서화 — 1인 개발에서도 실무 프로세스 유지

## E. 자소서 활용 가이드

- 각 항목의 **상황→결정→근거** 3문장이 그대로 자소서 한 문단: "~한 상황에서 ~을 고민했고, ~라는 근거로 ~를 선택해 ~한 결과를 얻었습니다"
- 기술 역량 문항 → B(성능)·C(트러블슈팅) / 문제 해결 문항 → A-1, C-1 / 협업·프로세스 문항 → D(결정로그 운영)
- 면접 꼬리질문 대비: 각 결정의 "반대편 선택지를 안 고른 이유"까지 말할 수 있어야 함 (이 문서의 '고민' 항목이 그것)
