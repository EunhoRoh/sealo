# 07. 개발 원칙 (성능 중심)

작성일: 2026-07-07
배경: 트래픽이 몰려도 느려지지 않는 서비스. 그리고 이 원칙들 자체가 대기업 면접 단골 주제라 커리어 목적과도 일치.

## 백엔드 (Spring Boot / JPA)

### N+1 쿼리 방지 — 최우선 규칙
- 모든 연관관계는 **지연 로딩(LAZY)** 기본. `@ManyToOne`도 명시적으로 `fetch = LAZY`
- 목록 조회 시 연관 엔티티가 필요하면 **fetch join** 또는 `@EntityGraph` 사용
- 컬렉션 fetch join의 페이징 문제는 `default_batch_fetch_size: 100` (전역 설정 완료)으로 in-쿼리 배치 처리
- 조회 전용 화면(홈의 오늘 루틴 목록, 캘린더)은 엔티티 대신 **DTO 프로젝션**으로 필요한 컬럼만 SELECT
- 개발 중 콘솔 SQL 로그를 항상 켜두고, 화면 하나에 쿼리가 몇 개 나가는지 확인하는 습관

### JPA 설정 원칙 (적용 완료)
- `open-in-view: false` — OSIV 끄기. 커넥션을 뷰 렌더링까지 물고 있으면 트래픽 몰릴 때 커넥션 풀 고갈의 주범
- 쓰기 없는 서비스는 `@Transactional(readOnly = true)` 기본

### 그 외
- 조회 조건에는 **인덱스** 설계 (예: routine_log의 member_id + date 복합 인덱스 — 캘린더 조회용)
- 목록 API는 처음부터 페이지네이션
- 통계처럼 무겁고 자주 안 변하는 데이터는 v2에서 Redis 캐시 후보
- 부하 테스트(k6)는 출시 전 1회 — 포트폴리오 소재로도 좋음

## 프론트 (React Native / Expo)
- 서버 상태는 TanStack Query — 캐싱/중복요청 제거가 기본 제공되어 불필요한 API 호출 방지
- 리스트는 FlatList (가상화), 이미지 에셋은 사전 최적화
- 도장 애니메이션은 JS 스레드가 아닌 네이티브 드라이버(Reanimated) 사용
